* Project Overview

*PREANN* = Parallel Reinforcement Evolutionary Artificial Neural Networks

A framework for training neural networks through genetic algorithms with high-performance parallel implementations (C++, SSE2, CUDA).

*Key Features*:
- Configurable multi-layer neural networks with flexible architectures
- Genetic evolution (selection, crossover, mutation)
- Task-based fitness evaluation (games: Chess, Go, Reversi)
- Multiple implementations: Pure C++, SSE2, CUDA GPU acceleration
- Persistent training with automatic save/load

* Current Chess Implementation

** Neural Network Architecture

*Current Architecture*: 768→8→4→2→1 (feedforward only)
- Input: 768 neurons (8x8 board × 12 piece types)
- Hidden Layer 0: 8 neurons (BT_FLOAT buffer)
- Hidden Layer 1: 4 neurons (BT_FLOAT buffer)
- Hidden Layer 2: 2 neurons (BT_SIGN buffer for bipolar)
- Output: 1 neuron (BT_FLOAT buffer, position evaluation)

*Topology String*: =b768_f8_f4_p2_f1=

*Buffer Types*:
- BT_BIT (b): 1 byte per weight (binary values, memory efficient)
- BT_SIGN (p): 1 byte per weight (bipolar values)
- BT_FLOAT (f): 4 bytes per weight (full precision)

** Competitive Co-Evolution

*Design*:
- Fixed adversary within each training run
- Adversary updates between runs (best individual from previous run)
- Prevents champion degeneration through self-play
- Gradual co-evolution across training sessions

*Bootstrap*: New populations train against random opponent initially

*Expected Fitness Drop Between Sessions*:
- When adversary updates to best from previous session, fitness SHOULD drop
- This indicates evolution is working - the evolved network is harder than random
- Neural networks have deterministic strategy (even if randomly initialized)
- Random player has NO strategy (different random moves each turn)
- Even after 2 generations, best evolved network >> random player
- If fitness doesn't drop on adversary update, evolution may not be working
- Population then adapts to new stronger adversary over subsequent generations

** Persistent Training

- Automatically saves to: data/populations/chess_persist.pop
- Loading behavior:
  - If file exists: loads, re-evaluates, enables competitive co-evolution
  - If file missing: creates new population with random opponent
- Resume training: just run the same command again

** CRITICAL: Evolved Populations Are Irreplaceable

*Why .pop files are precious*:
- Evolved populations represent hours/days of computational investment
- Unlike supervised learning, you CANNOT retrain and get the same results
- Evolutionary algorithms explore stochastically - interesting behaviors may NEVER emerge again
- Each .pop file is a unique research artifact

*Rules*:
- **NEVER delete data/populations/*.pop files**
- **NEVER run training without explicit user instruction**
- Treat .pop files as irreplaceable experimental data
- Testing/benchmarking should use COPIES or separate files

** Population Management

- Population size: 50 individuals
- Selection: 25 (tournament size 3)
- Crossover: 24 (uniform at weight level, must be even)
- Elite preservation: 1
- Mutation: 3 per individual, range ±0.5

* Network Topology Naming Convention

** Format

Topology strings describe network architecture in a compact format:
=<input>_<layer1>_<layer2>_..._<layerN>=

Each component: =<type><size>= with optional recurrent markers

** Buffer Type Prefixes

- =b= : BT_BIT (binary values, byte weights - memory efficient)
- =p= : BT_SIGN (biPolar/bipolar values, byte weights)
- =f= : BT_FLOAT (floating point values and weights)

** Recurrent Connection Markers (TODO - Not Yet Implemented)

*NOTE*: Recurrent notation is documented but not yet implemented in code.
Current topology strings do NOT include R/r markers.

When implemented, recurrent connections will be marked on BOTH layers involved:

- *Uppercase R<N>*: Marks where recurrent input is RECEIVED
  - Example: =b1024_R3= means layer 1 receives from layer 3
  - Position: After the underscore, before next layer

- *Lowercase r<N>*: Marks where recurrent output is SENT FROM
  - Example: =p64r1= means layer 3 sends back to layer 1
  - Position: Appended directly to the layer specification

** Layer Numbering for Recurrence

Layers are numbered from the FIRST HIDDEN LAYER (not input):
- Layer 0: Input buffer (768 neurons) - not a "real" layer
- Layer 1: First hidden layer (index 0 in code)
- Layer 2: Second hidden layer (index 1 in code)
- Layer 3: Third hidden layer (index 2 in code)
- Layer 4: Output layer (index 3 in code)

Note: Layer N cannot recur to layer N+1 or beyond (not implemented)

** Examples

*** Simple Feedforward Network
=b768_f1_f1_f1_f1=
- 768 BIT input
- Four 1-neuron FLOAT layers
- No recurrent connections
- Debug/minimal network

*** Production Chess Network
=b768_b1024_b512_p64_f1= (current - no recurrent markers yet)

When recurrent notation is implemented:
=b768_b1024_R3_b512_p64r1_f1=
- 768 BIT input
- Layer 1: 1024 BIT neurons, receives recurrent from layer 3 (R3)
- Layer 2: 512 BIT neurons
- Layer 3: 64 SIGN neurons, sends recurrent to layer 1 (r1)
- Layer 4: 1 FLOAT output

Reading the recurrence: Layer 3 (p64r1) → Layer 1 (b1024_R3)

*** Self-Recurrent Layer (hypothetical)
=b768_b1024_b512_p64r3_f1=
- Layer 3 sends recurrent connection to itself
- Provides single-step memory within layer

** Usage in Population Files

Population filenames include topology for clarity:
=chess_<topology>_P<size>.pop=

Examples:
- =chess_b768_b1_b1_p1_f1_P50.pop= (debug network, 50 individuals)
- =chess_b768_b1024_b512_p64_f1_P50.pop= (production network, current format)
- =chess_b768_b1024_R3_b512_p64r1_f1_P50.pop= (future: with recurrent markers)

This prevents accidental overwrites when experimenting with different architectures.

** Key Insights

*Recurrent Networks for Games*:
- Resetting state at game start (not between moves) is critical
- Provides consistent initial state for evolutionary learning
- Potential for emergent opponent modeling behavior
- Novel capability vs supervised learning approaches (e.g., Stockfish)

*Competitive Co-Evolution*:
- Fixed adversary per run = stable fitness landscape
- Adversary updates between runs = gradual improvement
- Self-play prevents unfair advantage and degeneration
- Critical: adversary must be tested against itself
