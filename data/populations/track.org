** Build System Fix (2025-11-17)

*Problem*: Makefile didn't pass FACT_FLAGS to test programs
- Test programs compiled with: `$(CXX) -c $< -o $@` (no flags)
- Factory compiled with: `$(CXX) $(FACT_FLAGS) -c $< -o $@` (has flags)
- Result: Test programs never got -DCUDA_IMPL or -DSSE2_IMPL defines

*Fix*: Updated Makefile line 164 to include FACT_FLAGS:
```
build/test/%.o: src/test/%.cpp
	$(CXX) $(FACT_FLAGS) -c $< -o $@
```

*Verification*:
- `make cpp`: Compiles with `-DCPP_IMPL`
- `make cuda`: Compiles with `-DCPP_IMPL -DCUDA_IMPL`
- Runtime logging now correctly shows implementation type
- C++ version: "Using C++ implementation (IT_C)"
- CUDA version: "Using CUDA implementation (IT_CUDA_OUT)"

*Performance Benchmark Results*:

Test 1: Minimal network (768→1→1→1→1, 2 generations, 50 individuals):
- C++ (IT_C): 45.3s wall time (19.3s reported)
- CUDA (IT_CUDA_OUT): 3m0.4s / 180.4s wall time (77.6s reported)
- **Result**: CUDA is ~4x SLOWER for tiny networks (GPU overhead > parallel benefit)

This confirms:
- Implementations are now correctly selected (IT_C vs IT_CUDA_OUT)
- Factory message printed: "Using C++ implementation" vs "Using CUDA implementation"
- Sessions 1-5 were all IT_C (explains identical performance)
- CUDA overhead not worth it for small networks/short runs

*Factory Design*:
- Build as many implementations as machine supports (cpp, sse2, cuda)
- Factory selects best implementation at runtime based on network size/architecture
- Small networks: C++ fastest (no GPU transfer overhead)
- Large networks: CUDA fastest (parallelism outweighs overhead)
- Deployment: Train on CUDA, deploy on SSE2/C++ depending on target hardware
